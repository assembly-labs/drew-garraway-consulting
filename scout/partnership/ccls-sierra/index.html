About Scout

A library assistant built by librarians, for libraries

Scout is a natural language discovery tool that helps patrons find what they need through plain conversation. Instead of navigating filters and keyword searches, patrons simply ask Scout what they're looking for — and get helpful, librarian-quality answers grounded in your actual catalog data.

Scout was built by Drew Garraway, a Reference Librarian at Tredyffrin Public Library, and Kenny Allen, a former librarian. We built Scout because we saw first-hand how patrons struggle with traditional catalog interfaces — and how much time staff spends helping people find things that should be easy to discover.

To work, Scout needs to connect to the library's catalog system. That's where the Sierra REST API comes in — it's the bridge that lets Scout search the real CCLS catalog and show patrons real-time availability.

Why it matters

Value for everyone in the library

👤

For Patrons

Ask in plain language — "I need a good audiobook for a road trip" or "Do you have anything about starting a small business?" — and get personalized, conversational recommendations with real-time availability. No more decoding boolean search or navigating the OPAC.

📚

For Librarians

Fewer repetitive "do you have this?" questions at the desk means more time for the meaningful work — reader's advisory, programming, and community engagement. Scout handles the catalog lookups so staff can focus on what humans do best.

🏛

For Libraries

A modern, accessible discovery experience that meets patrons where they are — on their phones, at home, any time of day. Increases catalog usage, surfaces underutilized collections, and demonstrates innovation to stakeholders and funders.

Accessibility & Inclusion

A catalog that works for everyone

Traditional library catalogs assume patrons can read English fluently, see small text, and understand library jargon like "OPAC," "bib record," and boolean operators. That's a barrier for a lot of the people libraries serve most. Scout removes those barriers.

🌐

ESL & Multilingual Patrons

Patrons can ask questions in their own words — imperfect English, mixed languages, conversational phrasing. Scout understands intent, not just exact keywords. A patron who types "book for learn English at home" gets useful results, not zero hits.

👁

Low Vision & Blind Patrons

Scout's conversational interface works naturally with screen readers — no complex navigation menus, dropdown filters, or tiny clickable elements to fight with. Patrons get clear, spoken answers instead of dense results pages.

🤝

Digital Literacy Gaps

Many patrons — older adults, new library users, people without tech experience — find catalog search intimidating. Talking to Scout feels like asking a librarian at the desk, not operating a database. That's the point.

Technical Detail

How Scout connects to Sierra

Architecture Overview

Scout acts as a read-only client to the Sierra REST API (v6). It authenticates via OAuth 2.0 Client Credentials, then makes standard GET requests to search and retrieve catalog data. The natural language layer interprets patron questions, translates them into API calls, and presents results conversationally.

Patron asks: "Do you have books about beekeeping?" │ ▼ Scout extracts intent + keywords │ ▼ Sierra API GET /v6/bibs/search?text=beekeeping │ → returns matching bib records ▼ Sierra API GET /v6/items?bibIds=[...] │ → returns availability per branch ▼ Scout formats natural language response │ ▼ Patron sees: "I found 3 books about beekeeping. The Beekeeper's Handbook is available at Tredyffrin right now..."

Sierra API Endpoints We'll Use

All requests are read-only (GET). Scout never creates, modifies, or deletes any records.

Endpoint Purpose Phase
GET/v6/bibs/search Keyword search across the catalog — title, author, subject, or all fields. Uses Sierra's Advanced Word Search index with relevance ranking. MVP
GET/v6/bibs/{id} Retrieve full bibliographic record — title, author, ISBN, material type, subjects, publication year, MARC data. MVP
GET/v6/items Get item-level detail — availability status (available, checked out, in transit), due dates, branch location, call number. MVP
GET/v6/branches List all CCLS branch locations with names and codes. Used to translate location codes into human-readable branch names. MVP
POST/v6/bibs/query Structured filtering with boolean logic (AND/OR) — for targeted searches by specific MARC fields, material type, or audience level. MVP
GET/v6/bibs/metadata Retrieve material type codes and labels (book, DVD, audiobook, etc.) for accurate format display. MVP

Authentication Flow

Scout authenticates using OAuth 2.0 Client Credentials — the standard server-to-server flow that Sierra supports. No patron credentials are involved in Phase 1.

1. Scout sends POST /v6/token with the client key and secret (Base64-encoded)
2. Sierra returns a bearer token valid for 60 minutes
3. Scout includes Authorization: Bearer <token> on all subsequent requests
4. Token is refreshed automatically before expiry — no manual intervention needed

API Roles Required

When creating the API key in Sierra Administration, these are the roles we need enabled:

✓ Bibs Read
✓ Items Read
✓ Branches Read
✓ Holdings Read (optional)

That's it — all read-only. No write access, no patron data access, no circulation operations. Phase 2 patron features (holds, checkouts) would require additional roles and patron-level OAuth, but that's a separate conversation down the road.
