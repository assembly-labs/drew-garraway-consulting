## Design Direction Summary

TOMO is a voice-first BJJ training journal built on a bold, dark, editorial design language. The brand personality is "The Dedicated Training Partner" -- a knowledgeable, warm, direct purple belt who supports without patronizing. The visual system uses `#111111` as primary background with `#F5A623` gold accent, a three-font stack (Unbounded for headlines, Inter for body at weight 500+, JetBrains Mono for labels/data), and a 4px spacing grid. The app recognizes two distinct user states: exhausted post-training (session logging with 56-80px touch targets, 90-second tolerance) and relaxed browsing (journal, insights, profile with 44px targets). The product philosophy rejects gamification, hollow positivity, and coach replacement, favoring specificity, process-orientation, and honest reflection. Green = positive/wins, red = negative/losses, never swapped. Minimum font size 12px, minimum font weight 500. No emojis, only SVG icons.

## Severity 4: Catastrophic

### [S4] DA-001: Insights screen is built but completely inaccessible from the main app
**Heuristic:** Visibility of system status / Match between system and real world
**Location:** Navigation > MainTabNavigator.tsx
**Problem:** InsightsScreen exists as a fully-built feature (1065 lines) but is not wired into the tab navigator or any navigation path. Users who have 5+ sessions will never discover their weekly/monthly/quarterly AI insights because there is no way to reach the screen. This is TOMO's core differentiator -- hidden from every user. Already tracked as CR-001.
**Current:** MainTabNavigator has 3 tabs: Journal, Log (center +), Profile. InsightsScreen is imported nowhere.
**Recommendation:** Add Insights as the 4th tab, positioned to the right of Journal and left of the center Log button. Use `Icons.Zap` or `Icons.TrendUp` at size 24. Tab label: "Insights" in the same JetBrains Mono style.
**Effort:** Low
**Implementation:** Add a new `InsightsTab` screen to the Tab.Navigator in MainTabNavigator.tsx. The screen component already exists and is fully functional.

## Severity 3: Major

### [S3] DA-002: No back/cancel navigation from Entry phase of session logger
**Heuristic:** User control and freedom (emergency exit)
**Location:** SessionLoggerScreen > EntryPhase (line ~660)
**Problem:** When a user taps the center Log (+) button and lands on the Entry phase, there is no visible way to go back to the Journal tab without recording or typing. The only options are "Record" and "Type instead." The user must figure out they can tap a different tab. For exhausted users who accidentally tapped the wrong button, this creates confusion.
**Current:** Entry phase shows a title "Log Your Training", chip selectors, and action buttons. No back/cancel affordance.
**Recommendation:** Add a "Cancel" text button in the top-left corner, styled identically to RecordingPhase cancel (`styles.recordingCancelText`). On tap, navigate to JournalTab.
**Effort:** Low
**Implementation:** Add a cancel row at the top of EntryPhase, matching the existing `recordingCancelRow` pattern.

### [S3] DA-003: Review phase is overwhelming for exhausted users
**Heuristic:** Recognition rather than recall / Aesthetic and minimalist design
**Location:** SessionLoggerScreen > ReviewPhase (line ~907)
**Problem:** The Review phase shows 10+ editable sections in a single scrollable view: training mode, duration, notes, techniques, sparring rounds, submissions given, submissions received, injuries, instructor, session type, and voice transcript. This violates the "90-second window" first principle. An exhausted user who just voice-logged their session faces a wall of fields. Many will abandon rather than scroll through and verify everything.
**Current:** All 10+ sections rendered sequentially in a ScrollView with no progressive disclosure.
**Recommendation:** Restructure into two-tier layout: (1) Top section shows pre-filled summary card (mode, duration, techniques as tags, notes preview) with a single "Save Session" CTA at 56px height. (2) Below the fold, a "Review Details" expandable section for instructor, injuries, sparring details, session type, and transcript. Default collapsed.
**Effort:** Medium
**Implementation:** Wrap sections 6-10 (submissions, injuries, instructor, session type, transcript) in a collapsible section with a "Show more details" toggle. Keep notes, techniques, mode, and duration visible.

### [S3] DA-004: No back navigation in onboarding flow
**Heuristic:** User control and freedom
**Location:** Onboarding > AboutYouScreen, YourTrainingScreen, GetStartedScreen
**Problem:** Onboarding is forward-only (Welcome > AboutYou > YourTraining > GetStarted) with no back button on any screen. If a user selects the wrong belt on Screen 2, they must complete onboarding and edit from Profile. iOS swipe-back may work via native stack but there is no visible affordance.
**Current:** Each screen has a title and "Continue" footer. No back button.
**Recommendation:** Add back button (Icons.Back, size 22, color gray400) top-left of screens 2-4. Use `navigation.goBack()`. 44x44 touch target.
**Effort:** Low
**Implementation:** Add header row with Pressable back button to AboutYouScreen, YourTrainingScreen, and GetStartedScreen, below OnboardingProgressBar.

### [S3] DA-005: 14px font size used ~12 times but doesn't exist in the type scale
**Heuristic:** Consistency and standards
**Location:** Multiple files -- JournalScreen (`cardTopic`), InsightsScreen (`headerSubtitle`, `weeklyDateRange`), GetStartedScreen (`optionDescription`, `chatBubbleText`, `chatCursor`), YourTrainingScreen (`locationCardDesc`, `skipLinkText`, `loadingText`, `chipText`)
**Problem:** Design system defines type scale as 12 > 13 > 15 > 17 with no 14px step. Yet 14px appears ~12 times across 5+ files. This creates visual inconsistency.
**Current:** `fontSize: 14` used for secondary descriptions, card text, and chip labels.
**Recommendation:** Audit all 14px instances and remap: if user needs to read it, use 15px; if supplementary metadata, use 13px. Use semantic role of text to decide.
**Effort:** Medium
**Implementation:** Global find for `fontSize: 14` and remap per context.

**Heuristic:** Consistency and standards

**Heuristic:** Accessibility / Touch targets
**Location:** SessionLoggerScreen > EntryPhase > `textOnlyButton` (line ~748)
**Problem:** Fallback link in exhausted-user context relies on text padding alone. No explicit minHeight.
**Current:** Implicit touch target from text + padding.

**Effort:** Low

### [S2] DA-009: Offline banner uses hardcoded paddingTop: 54
**Heuristic:** Flexibility / Platform conventions
**Location:** NetworkError.tsx > OfflineBanner (line ~58)
**Problem:** Hardcoded approximation of status bar + safe area. Breaks on Dynamic Island devices (too short) and smaller notch devices (too tall).

**Recommendation:** Use `useSafeAreaInsets().top` from react-native-safe-area-context.
**Effort:** Low

### [S2] DA-010: Success message uses exclamation mark, borders on hollow positivity

**Current:** `'Session Logged!'` and count from profile + 1.

### [S2] DA-011: AuthScreen password field has no visibility toggle

### [S2] DA-012: Journal empty state has no direct action button

## Severity 1: Cosmetic

### [S1] DA-014: Inconsistent header casing ("INSIGHTS" vs "Journal")
**Heuristic:** Consistency and standards
**Location:** InsightsScreen headerTitle (line ~678) vs JournalScreen headerTitle (line ~359)
**Problem:** Insights uses all-caps with `letterSpacing: 1`, Journal uses title-case. Inconsistent navigational pattern.
**Recommendation:** Standardize to title-case ("Insights"). Unbounded for headlines, not uppercase labels.
**Effort:** Low

### [S1] DA-015: Stop Recording button lacks visual weight as primary action
**Heuristic:** Aesthetic and minimalist design
**Location:** SessionLoggerScreen > RecordingPhase > `stopButton`
**Problem:** Stop button appears secondary to pulsing mic animation. For exhausted users, it should be the dominant action.
**Recommendation:** Give stop button `backgroundColor: colors.gold` with `color: colors.black`.
**Effort:** Low

### [S1] DA-016: Processing phase "Skip to manual entry" has weak visibility
**Heuristic:** User control and freedom
**Location:** SessionLoggerScreen > ProcessingPhase (line ~892)
**Problem:** Gray link text at bottom during a potentially 150-second wait. Escape hatch should be more prominent.
**Recommendation:** Delay appearance by 10-15 seconds, then fade in with gold text color.
**Effort:** Low

### [S1] DA-017: Weekly pulse dots have no accessibility labels
**Heuristic:** Accessibility
**Location:** JournalScreen > WeeklyPulse > `pulseDot` (line ~376)
**Problem:** 10x10px dots with no accessibility annotations. Screen readers cannot interpret them.
**Recommendation:** Add `accessibilityLabel` to container: `"X of Y sessions this week"`. Mark individual dots `accessibilityElementsHidden={true}`.
**Effort:** Low

## Design System Gaps

### [DS-001] No 14px in type scale but used ~12 times
Either add 14px as a sanctioned token or migrate all instances to 13/15. Currently the codebase is inconsistent with its own spec.

### [DS-002] No reusable DismissibleTag component
Review phase and Session Detail both render technique/injury tags with close buttons, built inline each time rather than as a shared component.

### [DS-003] No named style for cancel text link
gray400 / Inter / 15px pattern appears on 4+ screens with inline styles. Should be `typography.cancelLink`.

### [DS-004] No animation timing tokens in design-tokens.ts
Design system documents animation durations but code uses magic numbers (200ms, 250ms, 300ms, 400ms, 600ms).

### [DS-005] pressedStyles tokens exist but are unused
Most pressables use inline `opacity: 0.7` or `opacity: 0.85`. Dark theme spec says lighten on press, not reduce opacity.

## What the App Does Well

1. **Exhausted-user awareness is structurally embedded.** 56px `entryChip` minimum height, voice-first pipeline, pre-filled defaults, generous "None" injury defaults, auto-dismiss success screen -- all show genuine understanding of the post-training user state. Not decorative empathy; structural.

2. **Brand voice is remarkably consistent.** Empty states, error messages, success messages, onboarding chat payoff all speak in the same voice: specific, grounded, never hollow. The coach deferral footer on Insights ("Your coach knows your game better than any app") is excellent trust-building.

3. **Gym picker flow is excellent.** Location soft-ask with branded explanation, animated nearby gym cards with distance, seamless fallback to text search, clear skip path. Better than most apps' gym/location onboarding.

4. **Loading states are comprehensive and layout-matched.** Every screen has a skeleton loader matching its actual layout (JournalSkeleton, SessionDetailSkeleton, ProfileSkeleton, InsightsSkeleton). Eliminates layout shift.

5. **Onboarding typewriter + chat payoff is strong.** Cycling phrases on Welcome, sequential typewriter chat bubbles on GetStarted with typing indicator. Personalized with user's name, belt, gym. Demonstrates immediate value.

6. **Production hardening is mature.** Double-tap guards, stale closure fixes, offline queue, pipeline timeouts, concurrent save protection, file leak fixes -- real production quality.

## Strategic Recommendations

### Top 3 Quick Wins
1. Wire InsightsScreen into the tab bar (15 min, surfaces the biggest feature)
2. Add back buttons to onboarding screens 2-4 (same pattern, low effort)
3. Add cancel to Entry phase (one pressable, matches existing patterns)

### Top 3 Strategic Improvements
1. Restructure Review phase into summary card + collapsible details with prominent Save
2. Build a lightweight Dashboard/Home so returning users see value before logging
3. Simplify the voice-preference entry path (tap Log > see mic > record > done)
