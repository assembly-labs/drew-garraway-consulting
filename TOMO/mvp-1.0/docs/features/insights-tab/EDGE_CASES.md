# FEAT-002: Edge Cases -- What Happens Before (and Between) Insights

**Created:** 2026-03-30
**Problem:** A weekly insight needs a full week of training data to say anything meaningful. What does the user experience during the days and scenarios where we can't generate one?

---

## The Core Tension

A user signs up Monday. They log their first session that night. They tap the Insights tab Tuesday morning. What do they see?

If it's an empty state that says "come back in 6 days," they never come back. If it's a fake insight based on one session, it'll feel hollow and generic. Neither is acceptable.

The solution: **the Insights tab must acknowledge every session without pretending to have insights it doesn't have.**

---

## Temporal Edge Cases

### 1. Day 1: First Session Just Logged

**Scenario:** User completes onboarding, logs their first session, taps Insights.

**What they see:**

A short, specific acknowledgment of what was captured. Not an insight -- a confirmation that TOMO is paying attention.

```
Mar 25

First session in the books. Gi class, 60 minutes at Alliance Paoli.

Your first weekly debrief will drop after a few more sessions. Each
one you log gives TOMO more to work with.
```

This is NOT generated by Haiku. It's a **client-side template** that reads from the session data directly. No API call. Instant.

**Template logic:**
```
Session 1: "First session in the books. {mode} {kind}, {duration} minutes
{at gym}. Your first weekly debrief will drop after a few more sessions.
Each one you log gives TOMO more to work with."
```

### 2. Days 2-6: Building Toward First Weekly

**Scenario:** User has 2-4 sessions but no complete week yet.

**What they see:**

Each session logged adds a line. The tab becomes a running receipt of the week so far, building anticipation for the debrief.

```
Mar 25 - 28

3 sessions so far this week. Gi classes, some guard work,
a few sparring rounds logged.

Your first weekly debrief is building. It'll land once you
close out this week.
```

**Template logic:** Client-side. Reads session count, unique techniques, total minutes, whether sparring was logged. Summarizes in 2-3 lines. No Haiku.

**Progress indicator:** Subtle, not a progress bar. Just the session count in the subtitle: "3 sessions this week" in the screen header.

### 3. End of Week 1: First Weekly Insight Generates

**Trigger:** Sunday night, or Monday morning when the user opens the app.

**What they see:** The first real weekly debrief. Typewriter animation plays. This is the "aha" moment.

**Minimum data for generation:** 1 session. But the quality scales with data:

| Sessions | What Haiku can say |
|----------|--------------------|
| 1 | Limited. Acknowledge the session, note what was logged, focus on the act of showing up. Don't try to find "patterns" in one data point. |
| 2 | Slightly better. Can note if they trained consecutive days, if techniques repeated, basic mode/kind. |
| 3+ | Real patterns emerge. Technique repetition, consistency vs target, sparring observations. This is where insights start earning the name. |

**Prompt instruction for low-data weeks:**
```
If this is a 1-session week, keep it brief. Acknowledge what they
logged without over-analyzing a single data point. Do not say "you
showed a pattern" or "you're developing" from one session. It's
fine to keep it to 1-2 short paragraphs plus a focusNext. Thin
data is better served with honest brevity than padded analysis.
```

### 4. Mid-Week (Between Insights)

**Scenario:** It's Wednesday. Last week's insight is there. They've logged 2 sessions this week. What does the tab show?

**What they see:**

Last week's debrief (static, already viewed) is still the primary content. Below it, a subtle "this week so far" section with session acknowledgments.

```
[Last week's debrief -- static, full text]

---

This week so far

2 sessions. Gi class Tuesday, nogi Thursday. Your next
debrief will land at the end of the week.
```

The "this week so far" section is client-side, not AI-generated. It keeps the tab alive between weekly debriefs.

### 5. Week with 0 Sessions (Skipped Week)

**Scenario:** User didn't train at all this week. Sunday comes. No data.

**What happens:** No weekly insight is generated. The tab shows the last available insight from the previous week(s).

**What they see:**

```
[Previous week's debrief -- static]

---

No sessions logged this week. The mat will be there.
```

One line. No judgment. No pressure. No "you missed your target!" language. The PSYCHOLOGICAL_PROFILES.md is explicit: do not pathologize normal training fluctuations.

**Edge function behavior:** Generation is not triggered (requires 1+ session). No empty insight is created.

### 6. Return After 2+ Week Gap

**Scenario:** User hasn't trained in 3 weeks. Logs a session. Opens Insights.

**What they see:**

The last insight from 3 weeks ago is still there (collapsed under "Earlier"). The current week shows the session acknowledgment:

```
Mar 25

Back on the mat. Gi class, 60 minutes.

Your next weekly debrief will build from here. Welcome back.
```

When Sunday comes and the weekly generates, the prompt knows about the gap (no prior week data to compare). The prompt instruction:

```
If priorWeekDelta is null or there's a 14+ day gap since the last
session, do not compare to "last week." Acknowledge they're back
without making the gap feel like a failure. "First week back" is
sufficient framing. Do not say "you've been away" or reference the
gap duration.
```

### 7. User Signs Up Mid-Week (Thursday)

**Scenario:** User signs up Thursday, logs Thu + Sat sessions. Sunday comes.

**What happens:** Weekly insight generates with 2 sessions of data. The prompt receives a partial week.

**Prompt handling:** The `week_start` is still Monday, but only Thu-Sun have data. The prompt sees `sessionsThisWeek: 2` and `totalSessionsAllTime: 2`. It knows this is their first week.

```
If totalSessionsAllTime <= 5 and this is the user's first weekly
insight, frame it as a first-week reflection. Don't judge frequency
against their target -- they may have signed up mid-week.
```

## Data Quality Edge Cases

### 8. Sessions with No Techniques Logged

**Scenario:** User voice-logs: "Went to class tonight, about an hour, gi." No technique names extracted.

**What Haiku can say:** Focus on attendance, duration, and training mode. Don't reference techniques at all.

**Prompt instruction:**
```
If techniquesDrilled is empty, do not mention techniques or say
"you didn't log techniques." The user may have trained without
detailed logging. Focus on what IS there: attendance, duration,
mode, sparring (if any).
```

### 9. Sessions with No Sparring

**Scenario:** White belt who only does class drills, no live rolling.

**What Haiku does:** Skips sparring entirely. No mention of submissions, rounds, or rolling. This is normal and fine -- especially for white belts in their first few months.

**Prompt instruction:**
```
If sparringRounds is 0 and submissionsGiven/Received are empty,
do not mention sparring at all. Do not say "you didn't spar this
week." Many practitioners -- especially newer white belts -- don't
spar regularly, and that's fine.
```

### 10. Minimal Sessions (Mode + Duration Only)

**Scenario:** 3 sessions, all logged as "Gi class, 60 min." No techniques, no sparring, no notes, no transcript.

**What Haiku can say:** Consistency only. "Three sessions this week, all gi. The habit is forming." That's genuinely all there is to say, and it's enough for a white belt.

**Prompt instruction:**
```
If the data is sparse (no techniques, no sparring, no notes), keep
the insight short. 1-2 paragraphs about consistency and showing up.
Do not pad thin data with generic advice. Brevity with sparse data
is better than length with nothing to say.
```

### 11. All Identical Sessions

**Scenario:** Every session is "Gi class, 60 min, closed guard" for 4 weeks straight.

**Week 1-2:** "Closed guard work again. The repetition is the point."
**Week 3-4:** The prompt needs to notice the repetition and evolve the observation. Not "you're drilling closed guard again" (obvious). Instead: "Four weeks of closed guard. That's depth, not stagnation." Or for a 3-4 stripe white belt: "You keep coming back to closed guard. Whether you chose it or just end up there, it's becoming a position you know."

**Prompt instruction:**
```
If the same techniques appear 3+ weeks in a row, don't just report
the repetition. Evolve the observation: week 1 = name it, week 2 =
note the repetition, week 3+ = reframe it as intentional depth or
ask (indirectly) whether it's intentional or habitual.
```

### 12. Only One Training Mode

**Scenario:** User only trains gi. Never nogi.

**What Haiku does:** Nothing. Don't mention nogi. Don't suggest they try it. That's their coach's call. Some gyms are gi-only.

## Generation Edge Cases

### 13. Edge Function Fails

**Scenario:** Anthropic API is down, timeout, or network error.

**Client behavior:** The Insights tab shows the last available insight. The "this week so far" section still works (client-side). No error state unless there are zero insights ever.

**Retry:** Next time the user opens the tab or pulls to refresh, generation is re-attempted if still eligible.

### 14. Haiku Returns Generic Output

**Scenario:** Despite rich data, Haiku produces "You had a good week of training. Keep it up."

**Mitigation:**
- The system prompt is explicit about no filler, no hollow positivity
- Post-generation validation: if all paragraphs are under 20 words and none reference specific data (technique names, session counts, sparring details), consider it low quality
- Phase 1: accept and ship (the prompt will improve with iteration)
- Phase 2: add a quality check that retries with higher temperature if output seems generic

### 15. First-Ever Insight (No Prior Week)

**Scenario:** First weekly debrief. No `priorWeekDelta` to compare against.

**Prompt handling:** When `priorWeekDelta` is null, the COMPARED TO LAST WEEK section is omitted entirely. The prompt focuses on this week only. No "up from last week" or "down from last week" language.

### 16. User Has FEAT-008 Data Missing

**Scenario:** User onboarded before FEAT-008 shipped. No birthday, gender, or expanded goals in their profile.

**Prompt handling:** All FEAT-008 prompt sections are conditional. If birthday is null, AGE CONTEXT is skipped. If gender is null, GENDER CONTEXT is skipped. If training_goals is null or the old format, WHY THEY TRAIN section falls back to generic framing. The insight still generates -- it's just less personalized.

## The "This Week So Far" Feature

This is the key addition. Between weekly insights, the Insights tab isn't dead -- it shows a lightweight, client-side summary of the current week's logged sessions.

### How It Works

- Client reads sessions from the current week (Monday-today)
- No API call, no AI generation
- Renders below the last weekly debrief (or as the primary content if no debriefs exist yet)

### What It Shows

**Template variants:**

1 session:
```
{date}

{Mode} {kind}, {duration} minutes{at gym}. {First session ever? "First session in the books." : "Session logged."} Your {weekly debrief will drop / next debrief will land} after {a few more sessions / the end of the week}.
```

2-3 sessions:
```
{date range}

{count} sessions so far this week. {summarize: modes, any techniques, sparring if logged}. Your next debrief is building.
```

4+ sessions:
```
{date range}

{count} sessions this week{, above/at/below your target}. {summarize}. Your next debrief is building.
```

0 sessions this week (but has previous debriefs):
```
No sessions logged this week. The mat will be there.
```

### Visual Treatment

Same message style as insights. Inter 15px gray300. No special formatting. Just text. The week header (JetBrains Mono 11px gray600) shows the date or date range. No "THIS WEEK SO FAR" label -- that's too dashboard-y. Just the dates.

A thin 1px #222 separator between the "this week" section and the previous debrief below it.

## Summary: What the User Sees at Every Stage

| Stage | Insights Tab Content | Source |
|-------|---------------------|--------|
| Day 1, 0 sessions | Empty state: "Your debriefs will appear here" | Client |
| Day 1, first session | Session acknowledgment + "first debrief building" | Client template |
| Days 2-6, 1-3 sessions | Running "this week so far" summary | Client template |
| End of week 1 | First weekly debrief (typewriter) | Haiku |
| Mid-week 2+ | Last debrief (static) + "this week so far" below | Haiku + client |
| Week with 0 sessions | Last debrief + "No sessions this week" | Client |
| After 2+ week gap | Last debrief (older) + session ack on return | Client template |
| Sunday/Monday | New weekly debrief (typewriter) | Haiku |

The tab is NEVER empty after the first session is logged. It always has something specific to show.
