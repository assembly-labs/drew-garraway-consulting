Question 1

Needs Discussion

This is an interesting one that I want to think through with you carefully.

Current architecture: Gather is designed as a pickup-first platform. This was a deliberate decision based on studying the failures of Good Eggs ($200M+ burned), Farmigo ($26M burned), and Webvan ($800M bankruptcy) — all of which tried last-mile fresh food delivery and couldn't make the economics work.

That said, I'm not saying "no." I did some research on how platforms like Square, Toast, and Shopify connect to delivery services. Here's what's possible:

Option A: Middleware Integration
Use a service like Deliverect or Cuboh that consolidates Uber Eats, DoorDash, and Grubhub into a single integration. This would add delivery as an optional toggle at checkout. The customer bears the delivery fee; vendor margins stay the same whether pickup or delivery. Implementation: 8-12 weeks, ~$5-15K setup.
The key question for you:
Is delivery a Phase 1 must-have, or can we nail the pickup experience first and explore delivery as a Phase 2 add-on? I'd advocate for proving the core model at Berwyn first, then layering in delivery for markets where it makes sense.

The good news: if we do add delivery, the vendor's income stays identical whether the customer picks up or gets delivery. The customer pays the delivery fee directly to DoorDash/Uber. We're not changing the fundamental economics.

Question 3
Multiple CH locations dropdown (Berwyn, Princeton, Allentown, etc.)
Already Planned

Yes, the platform supports this natively.

Gather is built as a multi-market platform from the ground up. Each location (Berwyn, Princeton, Allentown) would be a separate "Market" entity with:

  • Its own vendor pool (some vendors may overlap across locations)
  • Its own product catalog and availability
  • Its own pickup schedule and time slots

Customers would select their market at the start of shopping. They can switch between locations, and their cart is location-specific (you can't mix items from Berwyn and Princeton in one order).

For launch: We'll start with Berwyn only. When you're ready to add Princeton or Allentown, it's a configuration task — not new development. The infrastructure is already there.
Question 4
How do we make money? Hoping to set up service charges and/or transaction fees, not markup on products
Exactly as Planned

This is exactly how Gather is designed.

The revenue model is transaction-based, not markup-based:

  • 2.5% marketplace transaction fee on online orders
  • 1% on SNAP/EBT processing (when applicable)
  • Optional service fees configurable per market

Key principle: We never take inventory risk, never hold stock, never set prices. Vendors control their products completely. We facilitate; we don't intermediate.

Example order:
Customer pays: $50.00
Platform fee (2.5%): $1.25
Vendor receives: $48.75

Clean, simple math. Vendor sees exactly what they're paying.

This model protects you from the "Good Eggs problem" — when you're marking up products, you have to predict demand perfectly or eat the losses on unsold inventory. We skip that entirely.

Question 5
Order sheet info and Shelf-to-sheet — want items on pick lists to be in order with where they are kept in the warehouse
Partially Scoped — Enhancement Needed

The pick list functionality is in scope. Warehouse location sorting needs to be added.

Current spec includes:

  • Consolidated packing lists per vendor
  • Pick lists organized by product (not by order) for efficiency
  • Order management hub with fulfillment status

What we need to add: A warehouse_location field on each product so pick lists can be sorted by physical location.

Recommended location format:
ZONE-POSITION-SHELF-BIN (e.g., A-02-2-01)

Zone A = Room temp produce
Zone B = Cold/refrigerated
Zone C = Shelf-stable/packaged

Pick lists would then print in physical walking order: all Zone A items together, then Zone B, etc.

I did some research on warehouse picking optimization. For your scale (50-100 orders/day), the best approach is:

  • Batch picking by time slot: Group all orders for the 10:00-10:30 pickup window, pick all those items at once
  • Sort by location within batch: Staff walks the warehouse once, not back-and-forth
  • Then organize by order for customer pickup

This saves ~50% picking time compared to picking one order at a time.

What I need from you: A rough sketch of how your warehouse is organized (zones, aisles, refrigerated areas) so we can set up the location codes correctly.

Question 8
Inventory tracking tied to warehouse AND retail store POS — so online sales deduct from store inventory in real-time
Partially Scoped — POS Integration Needs Details

Real-time inventory sync is planned. POS integration depends on which system you use.

The platform is designed for real-time inventory management:

  • Single source of truth: One inventory count that all channels draw from
  • Soft reservation on cart add: When a customer adds 3 bottles of EVOO to their cart, those 3 are soft-reserved for 15 minutes
  • Hard deduction on order confirm: When they complete checkout, the inventory is deducted
  • Real-time sync: Updates propagate across all channels immediately

For the retail store POS integration, I did some research. Here's what's possible:

If you're using Square:
Square has mature inventory APIs with webhook support. When an in-store sale happens, Square fires a webhook to Gather, and we update the online inventory immediately. This is the best-case scenario — real-time, bi-directional sync.
If you're using Toast, Clover, or Lightspeed:
All have APIs, but webhook support varies. We can still sync, but it might be polling-based (every 60 seconds) rather than instant.

What I need from you: What POS system does the retail store use (or will use)? That'll determine the integration approach.

The scenario you described — "only sell 7 of 10 EVOO bottles if 3 were ordered online" — is exactly what we're building. The POS employee would see a notification (or the inventory would already be updated) showing only 7 available.

Question 11
Vendor profiles — don't show "market sold out" for vendors who only sell once a month. Activate their profile only when they upload products.
Easy to Add

Yes, we can handle this.

Two approaches:

Option A: Manual activation
Market manager enables/disables vendor profiles manually. When the monthly vendor is ready to sell, you flip the switch.

Option B: Auto-activation based on products (recommended)
Vendor profile automatically hides when they have zero products available. When they upload products for the month, the profile automatically appears. No manual intervention needed.

How it would work:
- Vendor uploads products on Monday
- Profile automatically becomes visible in marketplace
- After market day, products marked as sold/removed
- Profile automatically hides again
- No "Sold Out" messaging — just not visible

This is cleaner than showing "sold out" for vendors who aren't actively selling that week.
